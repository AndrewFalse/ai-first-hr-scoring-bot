# HR-скрининг бот — Архитектура и стратегия реализации

Ты - профессиональный Python-разработчик. Твоя задача реализовать продукт: Telegram бот для скоринга потенциальных кандидатов
на вакансию AI-first инженер. Система должна быть актуально нв 2026 и работать на российский сегмент.

## Стек

| Компонент | Выбор |
|-----------|-------|
| Telegram-бот | aiogram 3.x (Python) |
| LLM-скоринг | OpenRouter API (доступ ко всем моделям) |
| Хранение результатов | Google Sheets API |
| Валидация ссылок | GitHub API (PyGithub) |
| Деплой | Docker на локальном сервере |
| Конфиг и секреты | .env + python-dotenv |

Никаких лишних слоёв. Стек минимальный, но каждый компонент оправдан и решает конкретную задачу.

## Ключевые архитектурные решения

### 1. Адаптивные вопросы, а не линейный скрипт

Вопросы не одинаковые для всех. После каждого ответа бот анализирует контекст и может добавить уточняющий вопрос. Это показывает понимание того, как AI должен работать в продукте — не как скрипт, а как думающий инструмент.

Логика ветвления:
- Кандидат не упомянул валидацию AI-результатов → добавить уточняющий вопрос на эту тему
- Кандидат говорит про архитектурные решения → углубиться в декомпозицию задач
- Кандидат даёт поверхностные ответы → попросить конкретный пример

### 2. Скоринг с reasoning, а не с цифрой

Бот не пишет "7 из 10". Бот пишет, почему 7 из 10, цитирует слова кандидата и объясняет, чего не хватило. Это достигается через структурированный промпт в Claude, который возвращает JSON с полем reasoning.

Пример вывода кандидату:
```
Критерий «делегирование AI»: 8/10
Вы описали, как Claude генерировал архитектуру — это хорошо.
Но не упомянули, как проверяли результат. Это важный сигнал.
```

### 3. GitHub как источник сигналов, а не просто ссылка

Когда кандидат присылает ссылку на репозиторий, бот через GitHub API достаёт:
- README (есть ли документация)
- Количество коммитов (это реальный проект или один push)
- Основной язык
- Дату последнего коммита

Эти данные передаются в Claude вместе с ответами кандидата и учитываются при скоринге. Валидация мусора: если кандидат присылает не GitHub-ссылку, несуществующий или приватный репо — бот вежливо объясняет, что именно не так, и просит повторить.

### 4. Доступ для админа через secret-команду

Вместо простого пароля — двухшаговая активация:

1. Рекрутер вводит команду вида `/admin_<уникальный_хеш>`, который генерируется при деплое и прописывается в .env
2. Бот проверяет Telegram user_id: если он не в whitelist, даже правильная команда не даст доступ
3. Whitelist заполняется через отдельную команду при первом запуске: `/init_admin <secret_key>`, где secret_key — одноразовый токен из .env

Это стандартная практика для self-hosted инструментов без внешней авторизации. Хеш генерируется при деплое, а не зашит в коде.

## Pipeline

### Со стороны кандидата
```
/start
  → Приветствие + объяснение формата
  → Вопрос 1
  → Ответ → Claude анализирует контекст в фоне
  → Вопрос 2 (или уточняющий, если нужно)
  → ...
  → Запрос GitHub-ссылки
  → Бот валидирует репозиторий через GitHub API
  → Claude получает все ответы + данные репо → генерирует скоринг
  → Кандидату показывается итоговый скор с reasoning по каждому критерию
  → "Спасибо, мы свяжемся с вами в течение 3 дней"
```

Время прохождения: 7–12 минут.

### Со стороны админа
```
/admin_<hash>  (или /init_admin при первом запуске)
  → Главный экран: сколько прошло, средний скор, количество hot-кандидатов
  → /top — топ-3 кандидата с кратким объяснением
  → /candidate <id> — полная карточка: все ответы + reasoning скоринга
  → /export — выгрузка в Google Sheet
```

Hot-кандидат: суммарный скор выше порога (настраивается в .env) + живой репозиторий. При появлении — уведомление в отдельный Telegram-чат рекрутеров.

## Скоринг: три критерия

| Критерий | Что оценивается |
|----------|----------------|
| Делегирование AI | Умеет ли осознанно отдавать задачи AI и проверять результат |
| Декомпозиция | Думает ли перед тем, как промптить, или сразу бросается делать |
| Критичность | Понимает ли trade-offs, видит ли ограничения AI |

Каждый критерий: 1–10, с текстовым объяснением и цитатой из ответов. Итоговый скор — сумма трёх. Hot-порог по умолчанию: 21 из 30.

## База данных

**Драйвер:** asyncpg (прямой async PostgreSQL, без ORM)

**Схема:** `bot/db/init.sql` — выполняется автоматически при первом старте контейнера через `docker-entrypoint-initdb.d`.

### Таблицы

| Таблица | Назначение |
|---------|-----------|
| `admins` | Персистентный whitelist администраторов (заменяет in-memory `set[int]`) |
| `candidates` | Одна строка = одна сессия скрининга |
| `candidate_answers` | Все вопросы + ответы сессии (базовые и адаптивные от LLM) |
| `github_analyses` | Данные репозитория кандидата (one-to-one с candidates) |
| `scoring_results` | Итоговый скоринг LLM по трём критериям (one-to-one с candidates) |
| `followup_questions` | 2-3 вопроса для живого собеседования от LLM |

### Ключевые constraints

- `idx_candidates_one_active` — partial unique index: запрет двух активных `in_progress` сессий у одного `telegram_id`
- `chk_*_score` — CHECK: каждый критерий от 1 до 10, total от 3 до 30

### Логика повторного прохождения

- `in_progress` сессия существует → продолжить с того места
- `scored` сессия существует → "вы уже прошли скрининг, мы с вами свяжемся"
- Иначе → создать новую запись

### Структура `bot/db/`

```
bot/db/
├── __init__.py
├── pool.py                  # create_pool() → asyncpg.Pool (min=2, max=10)
├── init.sql                 # DDL-схема, монтируется в postgres-контейнер
└── repositories/
    ├── __init__.py
    ├── admins.py            # is_admin(), add_admin() с upsert
    ├── candidates.py        # create/get/update сессий, stats, top
    ├── answers.py           # add_question(), set_answer(), get_answers()
    ├── github.py            # upsert_github_analysis()
    └── scoring.py           # insert_scoring_result(), insert_followup_questions()
```

### Интеграция с aiogram

Pool создаётся в `on_startup`, закрывается в `on_shutdown`, передаётся через `dispatcher["db_pool"]`. Aiogram 3.x инжектирует его в хендлеры автоматически:

```python
async def cmd_start(message: Message, state: FSMContext, db_pool: asyncpg.Pool) -> None:
    ...
```

## Обоснование выбора вопросов

Стандартные вопросы про AI-инструменты выявляют осведомлённость, а не мышление. Разработчик, который перечислит Cursor, Claude и v0, ничем не отличается от того, кто прочитал статью на Хабре.

AI-first мышление — это конкретный паттерн: человек видит задачу и сразу думает, какую её часть можно делегировать, как проверить результат и где AI ошибётся. Поэтому все вопросы построены на конкретных примерах из опыта, а не на абстрактных знаниях. Это сложнее придумать, но легче оценить.

Адаптивный вопрос про итерацию промптов добавлен намеренно: если кандидат даёт поверхностные ответы, это либо нервозность, либо отсутствие реального опыта. Уточняющий вопрос помогает различить одно от другого.
